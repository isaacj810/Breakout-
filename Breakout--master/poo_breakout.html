🔌 Integración con Arcade Machine SDK

El juego fue refactorizado para funcionar como un módulo independiente dentro del proyecto académico Arcade Machine. El SDK implementa una arquitectura de loop único centralizado: el Core del sistema gestiona la ventana, el bucle principal y las llamadas por frame. El módulo Breakout actúa como un "invitado" que solo implementa los métodos de la interfaz.

Core (SDK)
→
handle_events(events)
→
update(dt)
→
render()
→
pygame.display.flip()

⚠️ El módulo nunca llama a pygame.display.set_mode(), pygame.display.flip(), pygame.event.get() ni crea su propio while. Todas las rutas de assets usan pathlib.Path (sin rutas absolutas). Resolución estándar: 1024 × 768.

Herencia

Permite que una clase hija reutilice atributos y métodos de una clase padre, evitando repetición de código. Con la integración del SDK se añadió un segundo nivel de herencia.

elements.py & powers.py main.py ← SDK
  • Paddle, Ball y Block heredan de GameObject.
  • PowerUp también hereda de GameObject.
  • BreakoutGame hereda de GameBase (clase del SDK) nuevo
  • Al llamar super().start(surface) y super().stop(), se reutiliza el ciclo de vida del SDK sin duplicar código.
# Nivel 1 — jerarquía interna del juego
class GameObject:
    def __init__(self, image, x, y): ...

class Paddle(GameObject): ...
class Ball(GameObject):   ...
class Block(GameObject):  ...
class PowerUp(GameObject): ...

# Nivel 2 — integración con arcade-machine-sdk
from arcade_machine_sdk import GameBase, GameMeta

class BreakoutGame(GameBase):
    def start(self, surface):
        super().start(surface)  # hereda ciclo de vida del SDK
        ...

Encapsulamiento

Cada clase oculta su estado interno y expone solo los métodos necesarios. Con el SDK, BreakoutGame agrupa y oculta todo el estado del juego detrás de atributos privados.

elements.py & game.py main.py — BreakoutGame
  • Ball controla speed_x / speed_y internamente en move().
  • Block gestiona su propio ciclo de vida en hit().
  • BreakoutGame encapsula el estado completo: _game_state, _menu, _game, _transitioning, etc. nuevo
  • El SDK solo puede llamar a handle_events, update y render; no accede al estado interno del juego.
class BreakoutGame(GameBase):
    def __init__(self, meta):
        super().__init__(meta)
        # atributos privados — solo accesibles internamente
        self._game_state    = "menu"
        self._menu          = None
        self._game          = None
        self._transitioning = False
        self._fade_alpha    = 0

    # El SDK solo conoce esta interfaz pública:
    def handle_events(self, events): ...
    def update(self, dt):        ...
    def render(self):            ...

Abstracción

Se modela solo lo esencial, ignorando detalles. Con el SDK se añade una abstracción de nivel superior: GameBase es una clase abstracta real (ABC) que define el contrato que todo juego debe cumplir.

elements.py arcade_machine_sdk — GameBase
  • GameObject define la idea general de "cualquier cosa del juego": imagen, posición y dibujo.
  • GameBase (SDK) define la idea general de "cualquier juego arcade": debe poder procesar eventos, actualizarse y dibujarse. nuevo
  • GameMeta abstrae los metadatos del juego (título, autores, tags) con un patrón Builder. nuevo
  • draw_overlay() y create_level() abstraen lógica compleja en funciones simples.
# ABC del SDK — contrato abstracto para cualquier juego
from abc import ABC, abstractmethod

class GameBase(ABC):
    @abstractmethod
    def handle_events(self, events): pass
    @abstractmethod
    def update(self, dt):        pass
    @abstractmethod
    def render(self):            pass

# Builder Pattern — abstrae la construcción de metadatos
metadata = (GameMeta()
    .with_title("Breakout")
    .with_description("Juego clásico arcade")
    .with_release_date("04/03/2026")
    .with_group_number(1)
    .add_author("Equipo Breakout"))

Polimorfismo

El mismo método se comporta de forma diferente según el objeto que lo llame. Con el SDK, BreakoutGame sobreescribe los métodos abstractos de GameBase con la lógica específica del Breakout.

elements.py & powers.py main.py — métodos del SDK
  • draw() está sobreescrito en Paddle, Ball, Block y PowerUp — cada uno lo ejecuta a su manera.
  • Block.draw() solo dibuja si self.alive == True; los demás siempre se dibujan.
  • BreakoutGame.update(dt) sobreescribe GameBase.update(): maneja estados del juego, físicas y transiciones. nuevo
  • BreakoutGame.render() sobreescribe GameBase.render(): dibuja el frame según el estado actual del juego. nuevo
# Polimorfismo 1 — draw() en los objetos del juego
class Block(GameObject):
    def draw(self, screen):
        if self.alive: super().draw(screen)

class PowerUp(GameObject):
    def update(self):
        self.rect.y += self.speed  # cae hacia abajo

# Polimorfismo 2 — métodos abstractos del SDK
class BreakoutGame(GameBase):
    def render(self):          # sobreescribe GameBase.render()
        surface = self.surface  # surface gestionada por el SDK
        if self._game_state == "menu":
            self._menu.draw(surface)
        elif self._game_state == "game":
            self._game.draw(surface)
